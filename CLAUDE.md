# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A browser-based graph/production-simulation game built with Svelte 5 + Vite + TypeScript. The player places "nodes" (buildings: House, Farm, Forest, Sawmill, Shop, Chest, Hunger — in that order in the node-selection menu, per `NODE_TYPES`) on a pannable/zoomable map (drag the ground to pan, mouse wheel or pinch to zoom), connects them with directed edges, and assigns a resource (Villager, Exhausted Villager, Carrot, Wood, Plank, Gold Coin, or Sun) to an edge to make units flow from source to target at a steady rate. Time can be paused/resumed with the space key (see `togglePause`) as well as the Time Control menu. Every node type's own inventory conversion happens on tick boundaries only (never partway through one): House converts exhausted villagers → villagers (capped at 10); Farm generates 1 sun/tick (capped at 5) and, independently, converts 1 villager + 1 sun → 1 exhausted villager + 1 carrot; Forest converts villagers → wood + exhausted villagers; Sawmill converts villagers + wood → planks + exhausted villagers; Shop can only hold one non-coin resource type at a time and automatically sells one unit/tick of whatever that is, for one gold coin each — there's no UI to choose what it sells, since "whatever it's holding" already is the choice; Chest similarly can only hold one resource type at a time, up to 100 units of it, but has no conversion at all — it's pure storage; Hunger starts with 50 carrot and can hold any amount of it ("food"), and every tick subtracts 1/10th of the map's *total* villager population (villager + exhausted_villager, summed over every node) from its own carrot count, unconditionally and without rounding — it can and will go negative/fractional if there isn't enough food anywhere (the UI truncates to an integer for display, but the stored value keeps full precision, so the 1/10th-per-tick drain is smooth rather than a once-every-10-ticks step). Shop and Farm are rate-limited to one unit per tick; House/Forest/Sawmill instead convert everything they're holding in one shot each tick.

## Commands

- `npm install` — install dependencies
- `npm run dev` — start Vite dev server (also runnable via `./run.sh`)
- `npm run build` — production build
- `npm run preview` — preview a production build
- `npm run check` — type-check (`svelte-check` on the app, `tsc` on the node/config side); there is no separate lint script and no test suite in this repo currently

There is no test runner configured — do not assume `npm test` exists.

## Architecture

### State: `src/lib/state.svelte.ts`
All game state lives in a single reactive class instance (`game`), built with Svelte 5 runes (`$state`) inside a `.svelte.ts` module — this is the pattern for reactive state outside of `.svelte` files and should be reused rather than introducing a separate store library. `game` is a singleton exported directly (no context/props plumbing); components import it and read/mutate it directly.

Key state and the invariants around it:
- `nodes` / `edges` / `travelers` — the graph data and in-flight resource units (`GraphNode`, `GraphEdge`, `ResourceTraveler` — core data-shape interfaces, all in `src/lib/types.ts`). `GraphNode.inventory` is a plain `Record<resourceId, number>`; Svelte 5's `$state` deep-proxies it, so mutating `node.inventory[id] += 1` in place is reactive without reassigning the array.
- `selectedNodeId` / `selectedEdgeId` — drive which floating `ContextMenu` is shown; mutually exclusive (see `GameMap.svelte`, and the cross-clearing in `clickNode`/`clickEdge`/`startConnectFrom`/`clearSelection`). There used to also be a `configuringNodeId` for Shop's "what should I sell" Configure panel; it was removed once Shop started auto-selling whatever single resource it holds instead of a user-picked one — if a future node type needs its own configuration UI, that's the mechanism (a second `<select>`-style `ContextMenu`, gated by a dedicated id) to reintroduce, not a generic reusable one to prematurely rebuild now.
- `connectingFromId` — non-null while the user is in "connect mode" after clicking a node's Connect button. The *next* node click (via `clickNode`) either completes the edge or, if the same node is clicked again, cancels connect mode. This state machine is centralized in `clickNode`/`clickEdge` on `game` — don't duplicate connect-mode logic in components.
- `speed` (0 / 1 / 4) — simulation speed multiplier consumed by the `requestAnimationFrame` loop in `App.svelte`, which calls `game.tick(dt)` every frame. `simTime` is the accumulated sim-seconds shown in the top-left counter (0.1s precision). Pausing (`speed === 0`) skips accumulation *and* skips resource spawning/movement (both are driven from `tick`) — it does not stop the RAF loop itself. Always change `speed` through `setSpeed()`, never assign it directly: `setSpeed` also updates the private `lastActiveSpeed` (Normal or Fast, whichever was picked last) whenever the new value is non-zero, and `togglePause()` (bound to the space key in `App.svelte`, guarded so it doesn't fire while a form control like the edge's resource `<select>` is focused) uses that to resume at whichever speed was active before pausing — pressing space while Fast is running toggles between Fast and pause, not Fast and Normal.
- `camera` (`{ x, y, zoom }`) — the pan/zoom view state; see the coordinate-system section below. Technically UI/view state rather than simulation state, but it lives on `game` anyway rather than as a separate store, consistent with this file's "single source of truth" convention and to avoid prop-drilling it into every component that needs to convert a coordinate (`GameMap`, `NodeView`, `NodeSelectionMenu`).

When adding new mutable game concepts (new resource behaviors, worker assignment, etc.), extend this class rather than creating parallel stores — components already assume a single source of truth.

### The simulation tick: `TICK_SECONDS`
`TICK_SECONDS` (`src/lib/types.ts`, currently `1`) is the single constant governing simulation cadence: how often a resource departs along an assigned edge, how long a resource takes to travel an edge (always exactly one tick, **regardless of the edge's on-screen length** — moving the source/target only changes the animated path, never the duration), and how often every node type's `process` runs (every tick, no exceptions — there is no per-type interval override; Hunger was briefly built with a 10-tick lump-sum subtraction and deliberately changed to a smoother every-tick, 1/10th-sized subtraction instead, precisely so it wouldn't need one). If a future node genuinely needs to act less often than every tick, do what Hunger does — divide the per-interval amount by the interval and still run every tick — before reaching for a new interval mechanism; that's simpler and avoids a whole extra field/branch for a single node's benefit.

### Resource flow: edges → travelers → inventory
An edge has `resourceId: string | null` (`null` = "None", the default — set via the dropdown in the edge's `ContextMenu`) and `lastSpawnTime`. Each `tick`, `spawnTravelers()` walks edges with a `resourceId` and, once per `TICK_SECONDS` of elapsed `simTime`, tries to move one unit from the source node's inventory into a new `ResourceTraveler` (a `while` loop, not `if`, so fast-forward/large `dt` catches up on missed spawns rather than dropping them). A spawn is skipped (but the interval still consumed, so cadence stays fixed) if the source has none of the resource, **or** if `canNodeAccept(target, resourceId)` says the target is full — this is what makes a full House (10 villagers) stop pulling more in. `advanceTravelers()` moves each traveler along by comparing `simTime` against `startTime + duration` (`duration` is always `TICK_SECONDS`, not distance-based); on arrival it calls `canNodeAccept` **again** before crediting the target's inventory, and simply drops the unit (doesn't re-queue it) if that now fails — this second check exists because the departure-time check alone isn't enough: two edges can each independently pass it and then both arrive at the same target in the same tick, jointly overflowing a capacity (or an exclusivity rule like Chest's/Shop's "one resource type only") that neither single departure would have violated on its own. `TravelerView.svelte` is purely a reactive read of this state (position derived from `game.simTime` and the edge's *current* endpoints each frame, no animation loop of its own) — do the same for any future per-frame visual.

`processNodes()` checks every node once per frame: if the node's type defines a `process` function and enough `simTime` has elapsed since `node.lastProcessTime` (a full `TICK_SECONDS`), it fires (again via a `while` loop for catch-up, so a big fast-forward `dt` still fires `process` once per tick actually elapsed, not once total; this is what keeps Hunger's per-tick 1/10th subtraction exact even under fast-forward, rather than losing precision to however coarsely `dt` happens to land). `typeDef.process(node, this.nodes)` also receives every node on the map, not just itself — needed for Hunger's global villager count, unused by everyone else (a function with fewer declared parameters is a valid `NodeTypeDef.process`, so existing single-argument `process`es didn't need to change). House/Forest/Sawmill conversions are bulk/instantaneous (all matching inventory converts in one call); Shop and Farm move at most a fixed small amount (1 unit) per call instead; Hunger subtracts a fractional, unrounded amount (map-wide villager count ÷ 10) from its own carrot every single tick, which is what makes the drain look continuous rather than stepped.

Deleting a node/edge must not silently destroy resources that are mid-flight on it. `removeEdge`/`removeNode` both call a shared `deliverTravelers(edgeIds, excludeTargetId?)` *before* actually removing the edges — for each affected traveler it calls a shared `tryDeliver(target, resourceId, amount?)` (the same helper `advanceTravelers` uses for a normal arrival) to credit its resource straight to the edge's target inventory, subject to `canNodeAccept`, and only then drops the traveler from `this.travelers`. `removeNode` passes the node's own id as `excludeTargetId` so travelers that were *heading toward* the node being deleted are dropped rather than delivered (there's nowhere left to deliver them to) — but travelers on that node's *outgoing* edges still land normally on whatever other node they were headed to. A new mutation that removes edges/nodes needs to route through `deliverTravelers` the same way, not just filter `this.travelers` directly, or it'll reintroduce silent resource loss. Similarly, any new code crediting a resource to a node's inventory should go through `tryDeliver` rather than re-writing the `canNodeAccept` + `inventory[id] = (inventory[id] ?? 0) + amount` pair inline.

### Node types: `src/lib/nodeTypes/`
Each placeable node kind is its own file (`house.ts`, `farm.ts`, `forest.ts`, `sawmill.ts`, `shop.ts`, `chest.ts`, `hunger.ts`), exporting one `NodeTypeDef` const (`id`, `emoji`, `label`, `color`, optional `initialInventory`, optional `capacities`, optional `canAccept`, optional `process(node, allNodes)`). **To add a new node type: write a new file here and add it to the array in `index.ts` — nothing else needs to change.** The node-selection menu, the map, inventory rendering, and the edge's resource dropdown all key off `NODE_TYPES`/`RESOURCE_TYPES` generically; there's no per-`typeId` branching anywhere in the components. `process` is optional precisely so a pure-storage node like Chest can omit it entirely — it's just a `capacities` (+ `canAccept`) object with no behavior.

`nodeTypes/index.ts` assembles `NODE_TYPES: NodeTypeDef[]` from the individual files and also hosts three functions that operate on the catalog (so they live next to it rather than in `types.ts`): `getNodeType(nodeOrTypeId)` (accepts either a placed `GraphNode` or a raw type-id string — the single lookup every other piece of code should go through, rather than re-writing `NODE_TYPES.find((t) => t.id === ...)` inline), `getNodeCapacity`, and `canNodeAccept`.

- **Capacities**: `capacities[resourceId]` defaults to `0` (cannot hold) if unlisted. `UNLIMITED_CAPACITIES` (`src/lib/resources.ts`, built once from `RESOURCE_TYPES` so new resources are automatically unrestricted everywhere by default) is what Forest, Sawmill, and Shop use verbatim. House and Farm have their own restricted `capacities` object: House caps `villager` at 10 (`exhausted_villager` uncapped, everything else implicitly `0`); Farm spreads `UNLIMITED_CAPACITIES` but overrides `sun` down to 5 (`capacities: { ...UNLIMITED_CAPACITIES, sun: 5 }`) — that's the pattern to follow for a node that's unrestricted except for one specific resource. Chest instead builds its `capacities` straight from `RESOURCE_TYPES` with every resource capped at the *same* number (100, `Object.fromEntries(RESOURCE_TYPES.map((r) => [r.id, 100]))`) — that's the pattern for "capped uniformly across every resource" rather than "unrestricted except one." If a future node type needs a from-scratch restriction (accepts almost nothing, like House), write the object by hand instead of spreading/mapping from the full resource list.
- **`canAccept` (acceptance rules that depend on the rest of the inventory)**: `capacities` alone can only express "count of resource X ≤ N," which can't express "only one resource type at a time" — accepting a second, different resource must be refused even while under the numeric cap. For that, a `NodeTypeDef` can define `canAccept(node, resourceId, amount)`, and `canNodeAccept()` in `nodeTypes/index.ts` calls it *instead of* the default capacity math whenever it's present. Two examples, both exclusive-storage but with different exemptions: Chest checks whether any *other* resource key already has a positive count, then falls back to the 100 cap; Shop does the same check but exempts `gold_coin` from it both ways (accepting more coin is always allowed regardless of what else is stored, and coin already present doesn't block accepting some other single good) — since a shop necessarily accumulates coin *alongside* whatever it's selling. If a future node needs a stateful/cross-resource rule like this, write a `canAccept` rather than trying to force it into `capacities`; if it's just a per-resource ceiling, `capacities` alone is still the simpler and preferred choice.
- **Process functions**: each node file defines its own private `process(node, allNodes)` (not exported, not named `processX` anymore — there's no naming collision risk since it's module-scoped per file) and puts it on its `NodeTypeDef`. `state.svelte.ts` never branches on `typeId`; it just calls whatever `process` the type provides, always passing `this.nodes` as the second argument (most types ignore it). House converts all `exhausted_villager` into `villager`, capped by remaining room under 10 (leftovers wait for a future tick, which is what keeps conversion itself from breaking the cap). Forest converts all `villager` into equal `wood` + `exhausted_villager`. Sawmill is a two-input conversion: it consumes `villager` *and* `wood` 1:1 (limited by `Math.min` of whichever is scarcer) to produce equal `plank` + `exhausted_villager`. Shop finds whichever single non-`gold_coin` resource key has a positive count in its own inventory (there's at most one, since its `canAccept` enforces that) and sells 1 unit of it per tick for 1 `gold_coin` — it does *not* read any user-set "what to sell" preference; the single resource it happens to be holding *is* the choice, which is also why Shop has no Configure button anymore. Farm's `process` does two unrelated things in the same call, in order: first it tops up `sun` by 1 (capped at 5, i.e. a source that produces without consuming anything), then — completely independently, using whatever `sun`/`villager` happen to be in inventory *after* that top-up — it does a Sawmill-style *bulk* two-input conversion (`Math.min` of whichever is scarcer, not rate-limited to 1/tick), `villager` + `sun` → `exhausted_villager` + `carrot`. Farm is the template to copy for any future "generates a resource from nothing, and also separately bulk-converts" node. **Production nodes (House, Farm, Forest, Sawmill) are never rate-limited per-conversion** — each conversion conceptually takes one tick, but any number of them happen in that same tick, in parallel, limited only by current stock (`Math.min` for two-input conversions, the full available count for one-input ones). Only Shop is deliberately rate-limited to 1 unit/tick (that's its whole "can't sell too fast" point) — don't copy Shop's single-unit shape for a new production node without a similarly explicit reason; copy Farm/Forest/Sawmill's bulk shape by default. Hunger is the one type that actually uses the `allNodes` parameter: it sums `villager + exhausted_villager` across *every* node on the map (not just itself), divides by 10, and subtracts that from its own `carrot` — every tick, with no rounding and no floor at zero. This is the template for any future node whose behavior depends on global map state rather than just its own inventory; it's also the only place in the codebase where a resource count is allowed to go negative or fractional (see the `NodeView` note below), and the reference example for "smooth a periodic effect out over N ticks by dividing the per-period amount by N and running every tick" rather than literally running every Nth tick.

`emoji` (on `NodeTypeDef`, defined in `types.ts`) is `string | string[]`. A single string renders as-is; an array renders through `EmojiIcon.svelte`, which stacks each layer centered on the others, with the *last* layer scaled down (`0.72em`) so it reads as sitting/growing on top of the base rather than fully covering it. This composite-layering was tried once on an earlier node (a brown-square-plus-carrot combo) and reverted in favor of a plain single-emoji string — no current node type uses the array form, but `EmojiIcon` still supports it for whenever a future icon actually wants it. `EmojiIcon` is used everywhere a node's icon appears (`NodeView`, and both the menu item and drag-ghost in `NodeSelectionMenu`) — a composite icon just needs an emoji array in that node's file, no per-component changes.

### Resources: `src/lib/resources.ts`
`RESOURCE_TYPES` is the catalog for transportable/storable resources: `villager` 🧑, `exhausted_villager` 🥱, `carrot` 🥕, `wood` 🪵, `plank` 🟫, `gold_coin` 🪙, `sun` ☀️. It drives both the edge's resource dropdown and the emoji used for inventory badges/travelers — adding a resource is adding an entry here plus, if it should be restricted anywhere, a `capacities` entry on that node type. `UNLIMITED_CAPACITIES` (see above) is derived from this list, so it stays in sync automatically as resources are added. Resource emoji and node emoji are separate namespaces and can coincide without it being a bug — e.g. `plank`'s resource emoji is `🟫`, which Farm's node icon (now `🌱`) briefly also used before being changed.

The edge's resource `<option>`s (`GameMap.svelte`) render `r.emoji` only, no `r.label` text, with a bare `-` for "no resource" (deliberate — keep it that way for new resources). This is currently the *only* resource-picking UI in the app — Shop used to have a second one (a "what should I sell" dropdown) but it was removed once Shop started auto-selling whatever single resource it's holding.

`NodeView`'s inventory badges (`inventoryEntries`) filter on `count !== 0`, not `count > 0` — this is deliberate so Hunger's carrot count is still visible once it goes negative (e.g. `-12🥕`) rather than silently disappearing the moment it crosses zero. The badge itself renders `Math.trunc(count)`, not the raw value — Hunger's carrot count is fractional every tick (1/10th-of-population subtractions), and `Math.trunc` (not `Math.round`/`Math.floor`) is what "show only the integer part" means for a value that can be negative (`Math.floor(-12.7)` is `-13`, which is not "the integer part"; `Math.trunc(-12.7)` is `-12`, which is). The *stored* `node.inventory` value keeps full floating-point precision always — only the display is truncated, so there's no accumulated rounding error from displaying it. Nothing else in the game currently produces a negative or non-integer inventory count, so this only visibly changes Hunger's behavior in practice, but don't revert either of these without checking Hunger first.

### Core data shapes & geometry: `src/lib/types.ts`
Home for the structural interfaces (`NodeTypeDef`, `GraphNode`, `GraphEdge`, `ResourceTraveler`) and geometry/timing constants that don't belong to any one catalog: `TICK_SECONDS`, `NODE_RADIUS` (also used by `getEdgeGeometry()`, shared by `EdgeView` and `TravelerView`, to shorten arrows/travel paths so they stop at the node's edge rather than its center — keep these two in sync if node sizing changes), and `EDGE_OFFSET`. This file intentionally has no dependency on `nodeTypes/` or `resources.ts` — it's the shared vocabulary those catalogs are built from, not a catalog itself.

### Coordinate system: world space vs. screen space
Node/edge/traveler positions (`node.x`, `node.y`, and everything derived from them) are **world coordinates** — they do not change when the user pans or zooms. `game.camera` (`src/lib/state.svelte.ts`: `{ x, y, zoom }`) is the sole mapping between world space and screen space (client pixels relative to `#game-map`'s own top-left, i.e. `mapEl.getBoundingClientRect()`), via `game.worldToScreen(x, y)` and `game.screenToWorld(x, y)`. The formula: `screen = world * zoom + camera.{x,y}`. Any new code that needs to convert between a pointer event's `clientX`/`clientY` and a node's world position must go through these two methods — do not assume 1:1 anymore.

`GameMap.svelte` renders two layers inside `#game-map`:
- **`.world`** — a child div with inline `transform: translate(camera.x, camera.y) scale(camera.zoom)` and `transform-origin: 0 0` (must stay `0 0`; the camera math assumes scaling pivots on the top-left corner, not the CSS default center). Contains the edges `<svg>`, `NodeView`s, and `TravelerView`s, all still positioned with plain world-coordinate `left`/`top` — this layer is what actually gets panned/zoomed, and nothing inside it needed to change to support that. It has `pointer-events: none` so empty space passes through to the map background beneath it; `NodeView`'s `.node` explicitly sets `pointer-events: auto` back on to stay clickable/draggable (any new interactive element rendered inside `.world` needs the same override, or it'll silently stop receiving pointer events).
- **The overlay** — `ContextMenu` instances render as direct children of `#game-map`, *outside* `.world`, positioned in screen space via `game.worldToScreen(...)` computed in a `$derived`. This is deliberate: it keeps menu buttons a constant on-screen size regardless of zoom, rather than scaling (and becoming tiny/huge) with the graph. Any new floating UI anchored to a node/edge should follow this pattern — compute its screen position with `worldToScreen`, don't render it inside `.world`.

The dot-grid background is a `background-image` on `#game-map` itself (not `.world`), with `background-position`/`background-size` driven directly off `camera.{x,y}`/`camera.zoom` inline styles — CSS backgrounds tile infinitely for free, which a real DOM element panned via `.world` would not (its box is finite).

Because dragging a node needs a *delta* in world units but the pointer only reports screen pixels, `NodeView`'s drag math divides the raw screen-space delta by `game.camera.zoom` before adding it to the node's world position (see `onPointerMove`) — otherwise nodes would drift relative to the cursor at any zoom level other than 1. The click-vs-drag movement threshold, by contrast, is checked in *screen* pixels (a physical 4px twitch should count as "still a click" regardless of zoom).

### Pan & zoom gestures: `GameMap.svelte`
Panning (drag) and pinch-zoom both live in the map background's own pointer handlers (`onBackgroundPointerDown`/`Move`/`Up`), which only ever fire for gestures that actually started on empty map — every interactive child (nodes, edge hit-lines, context menus) calls `e.stopPropagation()` on `pointerdown`, so this code doesn't need to re-check `e.target` beyond the initial down.

- **State**: `activePointers: Map<pointerId, {x,y}>` tracks every currently-down pointer that started on the background, in map-local coordinates (not world — these are raw screen positions used purely for gesture math).
- **One active pointer** → pan: `game.panBy(dx, dy)` using the delta since that pointer's last recorded position.
- **Two active pointers** → pinch: on each move, compute the midpoint and distance between the two pointers *before* updating the moved one and *after*, then call `game.zoomToward(oldMid, newMid, camera.zoom * (newDist/oldDist))`. Passing different "anchor" (old) and "target" (new) screen points in the same call is what lets a single pinch pan and zoom simultaneously — a plain "zoom centered on a fixed point" (used by wheel zoom, `game.zoomAt`) is just `zoomToward` with the same point for both.
- **Click-vs-gesture**: `gestureStart`/`gestureMoved` distinguish "the user tapped empty space" (→ `game.clearSelection()`) from "the user panned/pinched" (→ do nothing on release). A second finger touching down always immediately sets `gestureMoved = true` (a pinch is never a tap), while single-finger movement only sets it once total displacement from `gestureStart` exceeds ~4px — mirroring the same threshold convention `NodeView` uses for node drags.
- **Wheel zoom** (desktop): `onwheel` calls `e.preventDefault()` (this is a plain, non-passive listener, so that's safe) and zooms exponentially (`Math.exp(-deltaY * 0.001)`) centered on the cursor via `game.zoomAt`.
- Zoom is clamped to `[0.3, 3]` inside `zoomToward`/`zoomAt` (`MIN_ZOOM`/`MAX_ZOOM` in `state.svelte.ts`) — always go through those methods rather than writing `camera.zoom` directly, or the clamp gets bypassed.

### Drag & pointer-event conventions
All dragging (moving placed nodes, spawning new ones from the bottom menu, and now panning/zooming the map) uses raw Pointer Events + `setPointerCapture`, not the HTML5 Drag-and-Drop API — this is deliberate, to get uniform mouse/touch/pen behavior for mobile support. Conventions to preserve:
- Distinguish a click from a drag via movement threshold (~4px, in screen pixels), not via a separate click handler (see `NodeView.onPointerMove` and `GameMap`'s `gestureMoved`) — a plain `onclick` would also fire after a drag.
- Elements that need to be clickable/draggable independently of the map background call `e.stopPropagation()` on `pointerdown`/`pointerup` so `GameMap`'s background handlers (pan/zoom/clear-selection) don't fire. Any new interactive element placed on the map needs the same guard.
- Draggable elements set `touch-action: none` in CSS; this is required for touch dragging (and for pinch-zoom, which we implement ourselves) to work — otherwise the browser intercepts the gesture for its own native scroll/zoom.

### Edges: `src/lib/EdgeView.svelte`
Each edge renders two overlapping SVG `<line>`s inside `GameMap`'s single `<svg>` overlay: an invisible wide-stroke "hit" line (`pointer-events: stroke`) for easy click targeting, and the visible arrow line with a `marker-end` arrowhead. The `<svg>` overlay itself is `pointer-events: none` so it doesn't block map background clicks; individual lines re-enable pointer events explicitly. Two `<marker>` defs (`#arrowhead`, `#arrowhead-selected`) live once in `GameMap`, not per-edge.

When both `A→B` and `B→A` edges exist between the same pair of nodes, each is pushed sideways by `EDGE_OFFSET` px (perpendicular to its own travel direction, via `getEdgeGeometry(source, target, offset)`) so the two lines don't overlap and stay independently clickable — the direction of the perpendicular is chosen so the pair reads as a clockwise loop. `EdgeView` and `TravelerView` each call `game.hasReverseEdge(sourceId, targetId)` and only pass a nonzero offset when it's true, so a lone edge between two nodes stays perfectly straight. `TravelerView` must apply the identical offset or the traveling icon will visibly drift off its edge's line. `game.hasReverseEdge` is a general "is there an edge running the opposite direction between these two nodes" query — it isn't specific to rendering, so a future feature that also cares about edge pairs (e.g. a bidirectional-flow node) should call it rather than re-deriving the same `.some(...)` check.

### Context menus
`ContextMenu.svelte` is a generic floating menu positioned via absolute `x`/`y` map-local coordinates, rendered conditionally by `GameMap` based on `selectedNodeId`/`selectedEdgeId`. It takes `items` (`label`, `title`, click handler, optional danger styling) for the button row, plus an optional `children` snippet rendered above the buttons for arbitrary controls — the edge's resource `<select>` is passed this way rather than adding a new menu component. Prefer extending the `children` snippet or `items` array at the call site in `GameMap.svelte` over building a second menu component.

Menu item `label`s are emoji, not text (➡️ Connect, 🗑️ Delete) — kept legible via the separate `title` field, which becomes both the native `title` tooltip and `aria-label` on the button. When adding a new action, pick a single representative emoji for `label` and put the real word in `title`, rather than reverting to text labels.

### Styling
Dark theme only, CSS custom properties defined once in `src/app.css` (`--bg`, `--panel`, `--accent`, etc.) and reused via `var(--x)` in component `<style>` blocks, including inside SVG (`stroke: var(--accent)`). Otherwise no CSS framework/utility classes — plain scoped Svelte `<style>` per component, with one deliberate exception: `.node-circle-fill` (global, in `src/app.css`) is the radial-gradient circle background keyed off a `--node-color` custom property, shared by `NodeView`'s `.node`, and `NodeSelectionMenu`'s menu-tile `.emoji` and drag `.ghost` — three places that needed byte-for-byte the same gradient. Add the class name alongside the element's own scoped class rather than copy-pasting the gradient again if a fourth place ever needs it.
